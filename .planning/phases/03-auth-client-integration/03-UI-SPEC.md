# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3: Auth Client Integration.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Web tool | Plain Tailwind CSS — to be installed in Phase 3 setup (not yet in apps/web/package.json) |
| Web component library | none (hand-rolled utility-class components) |
| Mobile tool | React Native Paper (MD3) — to be installed in Phase 3 setup (not yet in apps/mobile/package.json) |
| Mobile component library | React Native Paper v5 |
| Icon library | none for this phase |
| Web font | System font stack (no custom font this phase) |
| Mobile font | System default (RN Paper default) |

**Visual hierarchy:** Primary visual anchor is the centered form card. The page/screen heading (Sign in / Create account) is the first visual element the eye lands on; the submit button (Sign in / Create account) is the primary CTA and the dominant interactive element. All other content (links, error banners) is subordinate.

**Note to planner:** Phase 3 setup task must install `tailwindcss`, `postcss`, and `autoprefixer` in `apps/web`, and `react-native-paper` and `react-native-safe-area-context` in `apps/mobile`.

## Spacing Scale

Logical roles map to platform-specific values. Web uses Tailwind utility classes. Mobile uses React Native Paper's theme spacing or inline style values.

| Token | Value | Web (Tailwind) | Mobile (RN StyleSheet) | Usage |
|-------|-------|----------------|------------------------|-------|
| xs | 4px | `p-1` / `gap-1` | 4 | Icon gaps, inline padding |
| sm | 8px | `p-2` / `gap-2` | 8 | Compact element spacing, input inner padding |
| md | 16px | `p-4` / `gap-4` | 16 | Default element spacing, form field gaps |
| lg | 24px | `p-6` / `gap-6` | 24 | Section padding, card padding |
| xl | 32px | `p-8` / `gap-8` | 32 | Layout gaps, form container padding |
| 2xl | 48px | `py-12` | 48 | Major section breaks |
| 3xl | 64px | `py-16` | 64 | Page-level vertical spacing |

Exceptions: Touch targets (buttons, form submit) must be minimum 44px tall on mobile. Use `minHeight: 44` in RN styles.

## Typography

Logical roles map to platform-specific scales. Web values are px sizes via Tailwind. Mobile values map to RN Paper MD3 type scale tokens.

| Role | Size | Weight | Line Height | Web (Tailwind) | Mobile (RN Paper MD3) |
|------|------|--------|-------------|----------------|----------------------|
| Body | 16px | 400 (regular) | 1.5 | `text-base font-normal leading-relaxed` | `bodyLarge` |
| Label | 14px | 600 (semibold) | 1.25 | `text-sm font-semibold leading-tight` | `labelLarge` |
| Heading | 24px | 600 (semibold) | 1.2 | `text-2xl font-semibold leading-tight` | `headlineSmall` |
| Display | 32px | 600 (semibold) | 1.1 | `text-3xl font-semibold leading-none` | `headlineMedium` |

Exactly 2 font weights in use: 400 (regular) for body text, and 600 (semibold) for all heading and label roles. Display and Heading are differentiated by size alone (32px vs 24px), not weight.

## Color

A neutral-first palette appropriate for a functional marketplace tool. No decorative color this phase.

| Role | Web (Tailwind) | Mobile (RN Paper MD3 token) | Usage |
|------|----------------|-----------------------------|-------|
| Dominant (60%) | `bg-white` / `bg-gray-50` | `theme.colors.background` | Page background, form background |
| Secondary (30%) | `bg-gray-100` / `border-gray-200` | `theme.colors.surfaceVariant` | Input fields, form card container, dividers |
| Accent (10%) | `bg-indigo-600` / `text-indigo-600` | `theme.colors.primary` | Submit buttons ("Sign in", "Create account") only |
| Destructive | `text-red-600` / `border-red-400` | `theme.colors.error` | Inline field validation errors, top-level form error banner |

Accent reserved for: primary form submit buttons only. Do NOT apply accent to links, labels, or secondary UI elements in this phase.

Destructive color reserved for: validation error text, error border on invalid inputs, and top-level non-field error messages.

## States

This section is required because AuthContext exposes `isLoading` (D-03) and the `/api/auth/me` call runs on mount (D-01).

| State | Web | Mobile |
|-------|-----|--------|
| `isLoading === true` (auth rehydration on mount) | Full-page centered spinner (`animate-spin` on an `<svg>` ring, 32px, indigo-600) | RN Paper `<ActivityIndicator>` centered on screen, `color={theme.colors.primary}` |
| Form submitting | Submit button disabled, label replaced with "Signing in..." or "Creating account..." | RN Paper `<Button loading={true}>` with disabled state |
| Auth resolved, user present | Redirect to `/dashboard` (handled by middleware + D-04) | Redirect to home screen (handled by navigation guard) |
| Auth resolved, no user | Show login/register form | Show login/register screen |

## Error Rendering Contract

API returns structured field-map errors: `{ errors: { field: message } }` (per Phase 2 D-03).

**Inline field errors:**
- Render the message string from `errors[fieldName]` directly below the relevant input
- Style: `text-red-600 text-sm` (web) / `theme.colors.error` text via `<HelperText type="error">` (mobile)
- Show error border on the input: `border-red-400` (web) / `mode="outlined"` with error prop (mobile)

**Top-level fallback:**
- Trigger when: API returns non-200 but `errors` map has no recognized field keys (e.g., generic 500)
- Render a single error banner above the form
- Copy: "Something went wrong. Please check your details and try again."
- Style: `bg-red-50 border border-red-200 text-red-700 rounded p-3` (web) / RN Paper `<Banner>` with error icon (mobile)
