Output: Modified login, register, and /me routes that satisfy both transport modes without breaking existing web cookie behavior.

  Per D-02: /me must accept both web (httpOnly cookie) and mobile (Authorization: Bearer) tokens. The existing cookie parser (`split('token=')[1]`) is broken in multi-cookie scenarios — rewriting this code for D-02 requires replacing it with the NextRequest cookies API anyway.

  Replace the entire file with:
  ```typescript
  import { NextRequest, NextResponse } from 'next/server'
  import { verifyJwt } from '../../../../lib/auth'

  export async function GET(req: NextRequest) {
    const cookieToken = req.cookies.get('token')?.value ?? null
    const headerToken = req.headers.get('Authorization')?.replace('Bearer ', '') ?? null
    const token = cookieToken ?? headerToken
    if (!token) return NextResponse.json({ errors: { auth: 'missing' } }, { status: 401 })
    const payload = verifyJwt(token)
    if (!payload) return NextResponse.json({ errors: { auth: 'invalid' } }, { status: 401 })
    return NextResponse.json({ user: payload })
  }
  ```

  Key points:
  - `req` parameter type changes from `Request` to `NextRequest` — this enables `req.cookies.get()`
  - Cookie is tried first (web), then Authorization header (mobile) — per D-02 priority order
  - `?.replace('Bearer ', '')` strips the scheme prefix; do not trim further (JWT structure is preserved)
  - No revocation check needed here — Phase 2 revocation is in-memory and persists across the process; the same `verifyJwt` path still triggers it if Phase 2 embedded revocation in `verifyJwt`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client -> /api/auth/login | Token returned in JSON response body — mobile can capture it; this is intentional per D-02 |
| mobile client -> /api/auth/me | Authorization: Bearer header crosses the API boundary; token is untrusted until verifyJwt confirms signature |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | I | apps/web/app/api/auth/login/route.ts | mitigate | Token appears in response body — logs must not capture response bodies in production; this is a deployment concern, not a code change. Cookie remains httpOnly for web clients. Mobile stores token in SecureStore (plan 03-03/03-05), not localStorage. |
| T-03-02 | S | apps/web/app/api/auth/me/route.ts | mitigate | Dual-auth extraction tries cookie first — a browser attacker cannot inject an Authorization header via XSS if SameSite=Lax cookie is present. Header fallback only activates when no cookie is set (mobile context). |
| T-03-03 | T | apps/web/app/api/auth/me/route.ts | mitigate | `verifyJwt` validates JWT signature and expiry — forged or expired tokens return 401. The `replace('Bearer ', '')` strips only the scheme prefix; a malformed Authorization value produces an invalid token string that fails signature verification. |
</threat_model>
