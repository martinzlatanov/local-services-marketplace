<objective>
Extend the Phase 2 backend auth routes so they support both web (httpOnly cookie) and mobile (Authorization header + token in response body).

Purpose: Mobile cannot read httpOnly cookies — it needs the JWT in the response body to store in SecureStore, and must be able to prove identity via Authorization header. Without this, the mobile auth flow is blocked. Per D-02 from CONTEXT.md.
Output: Modified login, register, and /me routes that satisfy both transport modes without breaking existing web cookie behavior. /me now returns a full AuthUserDto queried from the database (not just JWT payload) so that `role`, `createdAt`, and `id` are always present with correct types.
</objective>

  Per D-02: /me must accept both web (httpOnly cookie) and mobile (Authorization: Bearer) tokens. The existing cookie parser (`split('token=')[1]`) is broken in multi-cookie scenarios — rewriting this code for D-02 requires replacing it with the NextRequest cookies API anyway.

  CRITICAL FIX: The JWT was signed with only `{ sub, email }`. Returning `verifyJwt(token)` directly means `role` and `createdAt` are absent from the payload, but `AuthUserDto` requires them. Both web and mobile AuthContext types assert `user.role` is `Role` — at runtime it would be `undefined`. Fix: after verifying the JWT, query the database by `payload.sub` and return a fully-typed AuthUserDto.

  Replace the entire file with:
  ```typescript
  import { NextRequest, NextResponse } from 'next/server'
  import { eq } from 'drizzle-orm'
  import { verifyJwt } from '../../../../lib/auth'
  import { db } from '../../../../lib/db/client'
  import { users } from '../../../../lib/db/schema'

  export async function GET(req: NextRequest) {
    const cookieToken = req.cookies.get('token')?.value ?? null
    const headerToken = req.headers.get('Authorization')?.replace('Bearer ', '') ?? null
    const token = cookieToken ?? headerToken
    if (!token) return NextResponse.json({ errors: { auth: 'missing' } }, { status: 401 })
    const payload = verifyJwt(token)
    if (!payload) return NextResponse.json({ errors: { auth: 'invalid' } }, { status: 401 })

    // Query DB for full user row — JWT payload only contains sub + email,
    // but AuthUserDto requires role and createdAt as well.
    const rows = await db.select().from(users).where(eq(users.id, Number(payload.sub))).limit(1)
    const user = rows[0]
    if (!user) return NextResponse.json({ errors: { auth: 'invalid' } }, { status: 401 })

    return NextResponse.json({
      user: {
        id: String(user.id),
        email: user.email,
        role: user.role,
        createdAt: user.createdAt.toISOString(),
      },
    })
  }
  ```

  Key points:
  - `req` parameter type changes from `Request` to `NextRequest` — this enables `req.cookies.get()`
  - Cookie is tried first (web), then Authorization header (mobile) — per D-02 priority order
  - `?.replace('Bearer ', '')` strips the scheme prefix; do not trim further (JWT structure is preserved)
  - After JWT verification, DB lookup by `users.id` using `eq` from drizzle-orm
  - If user row not found (deleted account), return 401 — same status as invalid token
  - `createdAt.toISOString()` ensures the string matches `AuthUserDto.createdAt: string`
  - Import paths: `../../../../lib/db/client` and `../../../../lib/db/schema` (db lives under lib/db/, not lib/db.ts)
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client -> /api/auth/login | Token returned in JSON response body — mobile can capture it; this is intentional per D-02 |
| mobile client -> /api/auth/me | Authorization: Bearer header crosses the API boundary; token is untrusted until verifyJwt confirms signature |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | I | apps/web/app/api/auth/login/route.ts | mitigate | Token appears in response body — logs must not capture response bodies in production; this is a deployment concern, not a code change. Cookie remains httpOnly for web clients. Mobile stores token in SecureStore (plan 03-03/03-05), not localStorage. |
| T-03-02 | S | apps/web/app/api/auth/me/route.ts | mitigate | Dual-auth extraction tries cookie first — a browser attacker cannot inject an Authorization header via XSS if SameSite=Lax cookie is present. Header fallback only activates when no cookie is set (mobile context). |
| T-03-03 | T | apps/web/app/api/auth/me/route.ts | mitigate | `verifyJwt` validates JWT signature and expiry — forged or expired tokens return 401. DB lookup further validates user existence; a valid JWT for a deleted user returns 401. |
