## Color

Inherited from Phase 8 (see Phase 8 UI-SPEC for the full 60/30/10 table). Phase 9 extends the accent and status badge colors only.

**Accent (10%) — Phase 9 additions to the "reserved for" list:**

The following elements join the Phase 8 accent list (`theme.colors.primary`):
- Detail screen "Start Work" button (`mode="contained"`, `ACCEPTED` status branch)
- Detail screen "Finish Work" button (`mode="contained"`, `IN_PROGRESS` status branch)

Do NOT apply accent to status badges/chips — use the semantic container tokens below.

**Status badge colors (Phase 9 — new):**

| Status | Paper MD3 token | Usage |
|--------|-----------------|-------|
| ACCEPTED | `theme.colors.secondaryContainer` (chip background) + `theme.colors.onSecondaryContainer` (chip text) | "Accepted" chip on Active Jobs Card |
| IN_PROGRESS | `theme.colors.tertiaryContainer` (chip background) + `theme.colors.onTertiaryContainer` (chip text) | "In progress" chip on Active Jobs Card |

Rationale: MD3 container tokens are muted but semantically distinct. They do not compete with the accent (`primary`) which is reserved for primary CTAs, nor with `error` which is reserved for destructive actions. Both meet WCAG AA contrast against the `surfaceVariant` Card background.

## Component Inventory & State Matrix

### Active Jobs Tab (`/(app)/(tabs)/active-jobs`)

| Component | Library | Notes |
|-----------|---------|-------|
| `Appbar.Header` + `Appbar.Content` | Paper | Title "Active Jobs" |
| `FlatList` | RN | `data={jobs}`, `keyExtractor={j => j.id}`, `ItemSeparatorComponent` = 16px gap. Mirror `feed.tsx` exactly. |
| `RefreshControl` | RN | `refreshing`, `onRefresh`, tint = `theme.colors.primary`. Fetches from `GET /api/jobs/mine`. |
| `Card` (per job) | Paper | `mode="elevated"`; entire card pressable via `onPress` → `router.push('/(app)/jobs/' + id)` |
| `Card.Content` | Paper | Wraps: status `Chip`, category (`labelLarge`), description (2 lines, `bodyLarge`), metadata row (timeframe + cityArea, `labelLarge`) |
| `Chip` (status) | Paper | `mode="flat"`, non-pressable (`onPress` omitted), placed above the category text within `Card.Content`. Chip style set inline per status token (see Color section). |
| `ActivityIndicator` | Paper | Centered during initial fetch; `color={theme.colors.primary}`, size `"large"` |

**Focus refresh mechanism:** Use `useFocusEffect` from `expo-router` (re-exports `@react-navigation/native`). Inside the callback, trigger a re-fetch from `GET /api/jobs/mine` when `token` is available. Pull-to-refresh is the manual fallback (D-04).

**State matrix:**

| State | Render |
|-------|--------|
| Initial fetch in flight | Centered `<ActivityIndicator size="large" />`, no list |
| Loaded, jobs.length === 0 | Empty state: heading "No active jobs" + body "Accept a job from the Feed tab to see it here." Centered, vertical padding `2xl` (48px). |
| Loaded, jobs.length > 0 | FlatList with Cards; each Card shows ACCEPTED or IN_PROGRESS chip; pull-to-refresh enabled |
| Tab focused (returning from detail or switching tabs) | `useFocusEffect` triggers re-fetch; list refreshes; COMPLETED jobs naturally disappear (not returned by API) |
| Pull-to-refresh active | `RefreshControl.refreshing = true`; list remains visible underneath |
| Fetch error (any non-200) | Top error banner above FlatList: "Couldn't load your active jobs. Pull down to retry." `theme.colors.error` text on `theme.colors.errorContainer` background. List shows last-loaded data (or empty). |

### Job Detail Screen (`/(app)/jobs/[id]`) — Phase 9 extensions

The Phase 8 state matrix for this screen (PENDING → Accept job path) is unchanged. Phase 9 adds two new CTA branches, rendered conditionally by `job.status`.

**Conditional CTA rule:**

```
if job.status === 'PENDING'  → "Accept job"      (Phase 8, unchanged)
if job.status === 'ACCEPTED' → "Start Work"       (Phase 9, new)
if job.status === 'IN_PROGRESS' → "Finish Work"   (Phase 9, new)
if job.status === 'COMPLETED'   → no CTA          (Phase 9, fallback)
```

**"Start Work" branch (ACCEPTED):**

| Component | Library | Notes |
|-----------|---------|-------|
| `Button` ("Start Work") | Paper | `mode="contained"`, full-width, `style={{ marginTop: 32 }}`, `contentStyle={{ minHeight: 44 }}`. Calls `PATCH /api/jobs/:id/status` with `{ status: 'IN_PROGRESS' }`. |

| State | Behavior |
|-------|----------|
| Idle | Button enabled, label "Start Work" |
| Submitting | Button `loading={true}` + `disabled={true}`, label "Starting..." |
| Success (200) | Stop loading; `router.back()` to Active Jobs list |
| 409 conflict | Show Snackbar "Job state changed. Pull down to refresh."; button returns to enabled |
| Other non-200 (network, 500) | Show Snackbar "Couldn't start work. Please try again."; button returns to enabled |

**"Finish Work" branch (IN_PROGRESS):**

| Component | Library | Notes |
|-----------|---------|-------|
| `Button` ("Finish Work") | Paper | `mode="contained"`, full-width, `style={{ marginTop: 32 }}`, `contentStyle={{ minHeight: 44 }}`. Calls `PATCH /api/jobs/:id/status` with `{ status: 'COMPLETED' }`. |

| State | Behavior |
|-------|----------|
| Idle | Button enabled, label "Finish Work" |
| Submitting | Button `loading={true}` + `disabled={true}`, label "Finishing..." |
| Success (200) | Stop loading; `router.back()` to Active Jobs list. Job disappears from Active Jobs on next focus-refresh (D-07). |
| 409 conflict | Show Snackbar "Job state changed. Pull down to refresh."; button returns to enabled |
| Other non-200 (network, 500) | Show Snackbar "Couldn't finish work. Please try again."; button returns to enabled |

**COMPLETED fallback (no CTA):**

If a provider navigates to a job with `status === 'COMPLETED'` (e.g., via deep link or stale navigation), render the job body fields normally but omit all CTA buttons. Show no error, no message — the job details are still useful. This is a silent graceful fallback; no `Text` label required.

**No confirmation dialog for "Finish Work":** The single-tap, no-confirm mobile aesthetic (established in Phase 3 and codified in Phase 8) applies to Finish Work. It is irreversible but not destructive (it advances a work contract, not deletes data). Do not add a confirmation dialog.

## Interaction Flows (Phase 9 additions)

### Flow 7: Active Jobs tab — browse and navigate to detail
1. Provider taps "Active Jobs" tab
2. `useFocusEffect` triggers `GET /api/jobs/mine`
3. List renders with ACCEPTED/IN_PROGRESS jobs, each showing a status Chip
4. Provider taps a Card → `router.push('/(app)/jobs/' + id)` — Stack push, slide-in
5. Detail screen fetches `GET /api/jobs/:id`; renders conditional CTA based on `job.status`

### Flow 8: Start Work success path
1. Provider opens an ACCEPTED job's detail; sees "Start Work" button
2. Taps "Start Work" → button enters loading state
3. `PATCH /api/jobs/:id/status` with `{ status: 'IN_PROGRESS' }` returns 200
4. `router.back()` returns to Active Jobs list
5. On next `useFocusEffect` re-fetch, job now appears with "In progress" chip

### Flow 9: Finish Work success path
1. Provider opens an IN_PROGRESS job's detail; sees "Finish Work" button
2. Taps "Finish Work" → button enters loading state
3. `PATCH /api/jobs/:id/status` with `{ status: 'COMPLETED' }` returns 200
4. `router.back()` returns to Active Jobs list
5. On next `useFocusEffect` re-fetch, job has disappeared (API returns ACCEPTED/IN_PROGRESS only per D-07)
6. Client dashboard receives WS broadcast of COMPLETED status (LIFECYCLE-04, already implemented in Phase 6)

### Flow 10: Status transition failure
1. Provider taps "Start Work" or "Finish Work"
2. Button enters loading state
3. API returns non-200 (409 conflict or network/5xx)
4. Snackbar shows appropriate message (see Copywriting Contract)
5. Button returns to enabled; provider can retry or tap back

## Accessibility Contract

Inherited from Phase 8. Phase 9 additions:

| Concern | Rule |
|---------|------|
| Active Jobs Cards | `accessibilityLabel="Job: {category} in {cityArea}, status {status}. Double-tap to view details."` and `accessibilityRole="button"`. |
| Status Chip | `accessibilityLabel="{status chip label}"`, `importantForAccessibility="no"` on the Chip itself if the Card's `accessibilityLabel` already announces status — avoids double-announcement. |
| "Start Work" button | `accessibilityLabel="Start work on this job"`, `accessibilityRole="button"`, `accessibilityState={{ disabled, busy: isSubmitting }}`. |
| "Finish Work" button | `accessibilityLabel="Finish this job"`, `accessibilityRole="button"`, `accessibilityState={{ disabled, busy: isSubmitting }}`. |
| Snackbar | Same rule as Phase 8 — Paper Snackbar uses `accessibilityLiveRegion="polite"` by default. |
