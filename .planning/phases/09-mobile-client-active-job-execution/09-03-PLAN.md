Purpose: The detail screen is the execution interface. By showing the appropriate button for the job's current status, providers can move work through its lifecycle. This consolidates all job interactions (accept, start, finish) in one place.

Output: Modified detail screen with conditional button rendering, new status transition handlers, and improved error messaging
</objective>

Add two new async handler functions to the component, placed after the existing `handleAccept` function.

**`handleStartWork` function:**
```typescript
async function handleStartWork() {
  if (!token || !job || isSubmitting) {
    return
  }

  setIsSubmitting(true)

  try {
    await updateJobStatus(token, job.id, JobStatus.IN_PROGRESS)
    router.back()
  } catch (err: any) {
    const errorMsg = err?.errors?.['status'] || "Couldn't start work. Please try again."
    setSnackbar(errorMsg)
  } finally {
    setIsSubmitting(false)
  }
}
```

**`handleFinishWork` function:**
```typescript
async function handleFinishWork() {
  if (!token || !job || isSubmitting) {
    return
  }

  setIsSubmitting(true)

  try {
    await updateJobStatus(token, job.id, JobStatus.COMPLETED)
    router.back()
  } catch (err: any) {
    const errorMsg = err?.errors?.['status'] || "Couldn't finish work. Please try again."
    setSnackbar(errorMsg)
  } finally {
    setIsSubmitting(false)
  }
}
```

**Handler pattern:**
- Check preconditions (`token`, `job`, not already submitting)
- Set `isSubmitting(true)` to disable button and show "Starting..." / "Finishing..." loading text
- Call `updateJobStatus(token, job.id, newStatus)` with the target status enum
- **On success:** `router.back()` to return to the list (per D-06). The Active Jobs list will refetch on tab focus (per Plan 02), showing the updated job or removing it if COMPLETED.
- **On error:** Extract error message from `err?.errors?.['status']` (backend validation error) or use generic fallback. Show in snackbar and stay on screen. User can retry or navigate back manually.
- **Finally:** Set `isSubmitting(false)` to re-enable the button

**Why:** These handlers follow the exact same pattern as `handleAccept` (error handling, loading state, navigation). The only differences are the API function (updateJobStatus vs acceptJob) and the target status. This consistency makes the code predictable and maintainable.

**Error handling notes:**
- Backend validates state machine transitions: ACCEPTED→IN_PROGRESS, IN_PROGRESS→COMPLETED. Invalid transitions return 400 with `{ errors: { status: 'invalid_transition_from_...' } }`
- Network errors also throw with a status code (parsed by `parseResponse`)
- Snackbar shows the error message so the user knows what went wrong (e.g., "invalid_transition_from_PENDING" if somehow the detail screen has stale state)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mobile UI → `PATCH /api/jobs/:id/status` | Untrusted job ID and status in body; endpoint validates that providerId matches user.id and status transition is legal |
| Router.back() → List navigation | Trusted; Expo Router handles navigation, no user input involved |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-07 | Tampering | updateJobStatus request body | mitigate | Backend validates (1) user owns job (providerId check), (2) transition is legal (state machine). Mobile sends correct status enum; backend rejects invalid transitions. |
| T-09-08 | Denial of Service | Rapid button taps | mitigate | `isSubmitting` flag disables button while request is in flight. User cannot submit duplicate requests. |
| T-09-09 | Elevation of Privilege | Non-provider changes job status | mitigate | `PATCH /api/jobs/:id/status` enforces `user.role === PROVIDER` (already implemented, Phase 4). Mobile UI only accessible to authenticated users on the PROVIDER path. |
