## Mobile API Client Pattern
**Existing helpers in `apps/mobile/lib/api.ts`:**
- `parseResponse<T>(res: Response)`: Parses JSON, throws error with status if not ok
- `getJobs(token, cityArea)`: Fetches `GET /api/jobs?cityArea=...` with Bearer auth
- `getJob(token, id)`: Fetches `GET /api/jobs/:id` with Bearer auth
- `acceptJob(token, id, version)`: POSTs to `/api/jobs/:id/accept` with Bearer auth
- All include error status in thrown error for caller detection (e.g., 409 handling in detail screen)

## Shared Types (from `packages/types/src/index.ts`)
- `JobDto`: Includes `id`, `status`, `version`, `category`, `description`, `timeframe`, `cityArea`, `clientId`, `providerId`, `createdAt`, `updatedAt`
- `JobStatus`: Enum or union: PENDING, ACCEPTED, IN_PROGRESS, COMPLETED
- `ApiSuccessResponse<T>`: `{ data: T }`
- `ApiErrorResponse`: `{ errors: Record<string, string> }`
- `UpdateJobStatusRequest`: `{ status: JobStatus; version?: number }`

Add two new async helper functions to `apps/mobile/lib/api.ts`, appended after the existing `acceptJob` function.

**`getMyJobs` helper:**
```typescript
export async function getMyJobs(token: string): Promise<JobDto[]> {
  const res = await fetch(`${API_BASE}/api/jobs/mine`, {
    headers: { Authorization: `Bearer ${token}` },
  })
  return parseResponse<JobDto[]>(res)
}
```
- Calls new `GET /api/jobs/mine` endpoint with Bearer auth
- Returns typed `JobDto[]` (provider's ACCEPTED + IN_PROGRESS jobs)
- Errors thrown by `parseResponse` include HTTP status, usable for error handling in UI

**`updateJobStatus` helper:**
```typescript
export async function updateJobStatus(token: string, id: string, status: string): Promise<JobDto> {
  const res = await fetch(`${API_BASE}/api/jobs/${id}/status`, {
    method: 'PATCH',
    headers: {
      Authorization: `Bearer ${token}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({ status }),
  })
  return parseResponse<JobDto>(res)
}
```
- Calls existing `PATCH /api/jobs/:id/status` endpoint (implemented in Phase 4)
- Sends `{ status: JobStatus }` in body
- Returns updated `JobDto` after status transition
- Errors thrown include HTTP status for caller error detection (e.g., 400 for invalid transitions)

**Why:** These helpers provide type-safe clients for (1) fetching the provider's active jobs and (2) updating job status. They follow the same pattern as `getJobs`, `getJob`, `acceptJob` — Bearer auth, error status propagation, typed returns.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → `/api/jobs/mine` | Untrusted token in Authorization header; endpoint must verify JWT and enforce PROVIDER role implicitly (by checking providerId match) |
| `/api/jobs/mine` → Database | Trusted; SQL via Drizzle ORM |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Tampering | `GET /api/jobs/mine` query params | mitigate | GET has no query params; no injection vector. Token verified via `getAuthenticatedUser` middleware. |
| T-09-02 | Information Disclosure | Provider lists other providers' jobs | mitigate | WHERE clause filters by `providerId = String(user.id)` — no data leakage. Each provider sees only their own. |
| T-09-03 | Denial of Service | `/api/jobs/mine` full table scan | accept | No filtering by status range; all-in-memory query is acceptable for provider's own job count (typically < 100 concurrent jobs per provider). |
| T-09-04 | Elevation of Privilege | Non-provider updates job status | mitigate | `PATCH /api/jobs/:id/status` existing route (Phase 4) already enforces `user.role === PROVIDER` and `job.providerId === user.id`. No new code path. |

</threat_model>
