<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser/mobile client -> auth endpoints | Untrusted credentials, roles, and cookies enter the API and must be validated before use |
| auth endpoints -> jwt cookie/session state | Signed session material is created and later re-read; the cookie must not expose the raw token to scriptable clients |
| auth endpoints -> database | User rows are written and looked up through the persistence layer; password hashes and unique email enforcement must survive this boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | S/T | apps/web/app/api/auth/register/route.ts | mitigate | Validate request bodies at the route boundary, clamp role to the shared enum, and reject duplicate emails with a database-backed unique constraint |
| T-02-05 | I | apps/web/app/api/auth/login/route.ts | mitigate | Issue JWTs only in an httpOnly, SameSite=Strict cookie, keep the token out of JSON responses, and use short-lived 15-minute expiry |
| T-02-06 | R/D | apps/web/lib/auth.ts | mitigate | Add JTI-based revocation in the auth utility and have the protected probe route reject revoked tokens before returning a user payload |
</threat_model>
