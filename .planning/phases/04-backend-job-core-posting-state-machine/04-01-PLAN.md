  <action>
    Add the Job model to `apps/web/lib/db/schema.ts` using Drizzle ORM pgTable.

    Import required types from drizzle-orm/pg-core: serial, varchar, text, timestamp, integer, pgEnum.

    Define a pgEnum for job categories:
    ```typescript
    export const jobCategoryEnum = pgEnum('job_category', [
      'PLUMBING', 'ELECTRICAL', 'CLEANING', 'GARDENING', 'MOVING', 'HANDYMAN', 'PAINTING', 'OTHER'
    ]);
    ```

    Define the jobs table:
    ```typescript
    export const jobs = pgTable('jobs', {
      id: serial('id').primaryKey(),
      status: varchar('status', { length: 32 }).notNull().default('PENDING'),
      version: integer('version').notNull().default(1),
      category: jobCategoryEnum('category').notNull(),
      description: text('description').notNull(),
      timeframe: varchar('timeframe', { length: 100 }).notNull(),
      cityArea: varchar('city_area', { length: 100 }).notNull(),
      clientId: varchar('client_id', { length: 320 }).notNull(),
      providerId: varchar('provider_id', { length: 320 }),
      createdAt: timestamp('created_at').defaultNow().notNull(),
      updatedAt: timestamp('updated_at').defaultNow().notNull(),
    })
    ```

    Update the JobDto import in packages/types to match: Ensure the `category` field type is `string` (since we're using enum in DB but string in TypeScript for simplicity across platforms).

    Add the relation: The clientId should reference users.email (or users.id if we had it as string). Since the existing users table uses serial id, we need to handle this properly. For now, store clientId as the user's email or id as string.

    Actually, looking at the existing schema, the users table has serial id (number). Let's update the jobs table to reference users.id properly:

    ```typescript
    clientId: integer('client_id').notNull().references(() => users.id),
    providerId: integer('provider_id').references(() => users.id),
    ```

    But wait, the JobDto in types has clientId: string. Let's keep it consistent - use integer for DB and convert to string in API responses.

    Actually, let's simplify: use varchar for clientId/providerId to match the JobDto string type, and store the user's id as string (convert number to string when storing).

    Final decision: Keep clientId and providerId as varchar in the database to match the JobDto interface.

    If drizzle-kit is not configured, create `drizzle.config.ts` in apps/web:
    ```typescript

      schema: './lib/db/schema.ts',
      out: './drizzle',
      driver: 'pg',
      dbCredentials: {
        connectionString: process.env.DATABASE_URL!,
      },
    } satisfies Config
    ```

    Then run the migration generation. The generated SQL should be reviewed and committed as `drizzle/0001_add_jobs.sql`.

    Note: Since we may not have drizzle-kit configured or database access, create a manual migration SQL file:

    Create `drizzle/0001_add_jobs.sql`:
    ```sql
    -- Create job category enum
    CREATE TYPE job_category AS ENUM ('PLUMBING', 'ELECTRICAL', 'CLEANING', 'GARDENING', 'MOVING', 'HANDYMAN', 'PAINTING', 'OTHER');

    -- Create jobs table
    CREATE TABLE jobs (
      id SERIAL PRIMARY KEY,
      status VARCHAR(32) NOT NULL DEFAULT 'PENDING',
      version INTEGER NOT NULL DEFAULT 1,
      category job_category NOT NULL,
      description TEXT NOT NULL,
      timeframe VARCHAR(100) NOT NULL,
      city_area VARCHAR(100) NOT NULL,
      client_id VARCHAR(320) NOT NULL,
      provider_id VARCHAR(320),
      created_at TIMESTAMP NOT NULL DEFAULT NOW(),
      updated_at TIMESTAMP NOT NULL DEFAULT NOW()
    );
    ```
