# Phase 8 â€” UI Design Contract

> Visual and interaction contract for Phase 8: Mobile Client â€” Job Discovery & Acceptance.
> Generated by gsd-ui-researcher. Mobile-only (Expo / React Native). Reuses tokens established in Phase 3 UI-SPEC.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (no shadcn â€” RN doesn't consume shadcn web registries) |
| Preset | not applicable |
| Component library | React Native Paper v5 (MD3) â€” already wired in `apps/mobile/app/_layout.tsx` |
| Icon library | `@expo/vector-icons` (Material Community Icons) â€” already a transitive dep of expo / expo-router; used as `<Appbar.Action icon="..."/>` and `<Tabs.Screen options={{ tabBarIcon }}/>` |
| Font | System default (Paper MD3 default) |

**Visual hierarchy per screen:**

- **Onboarding:** Heading ("Choose your service area") is the first anchor; the radio list is the dominant interactive area; "Save area" CTA at bottom is the only primary action.
- **Feed (Tab 1):** `Appbar` with current service area as title; the FlatList of `Card`s is the dominant content; each Card's category + cityArea is the visual anchor; the entire Card is the tap target (no in-Card buttons).
- **Settings (Tab 2):** Section list â€” current service area row is the anchor; "Log out" action is destructive and visually subordinate to "Edit area".
- **Job Detail:** Job category is the screen heading (largest type); the "Accept job" button at the bottom is the dominant CTA â€” no other primary action competes with it.

## Color

Reused from Phase 3 (Paper MD3 default theme â€” no custom theme this phase).

| Role | Paper MD3 token | Usage in this phase |
|------|-----------------|---------------------|
| Dominant (60%) | `theme.colors.background` | Screen background (all screens), FlatList container |
| Secondary (30%) | `theme.colors.surfaceVariant` | Card surface, Settings list rows, RadioButton row backgrounds, Snackbar surface |
| Accent (10%) | `theme.colors.primary` | "Accept job" button (Detail), "Save area" button (Onboarding), active tab indicator, RefreshControl tint, RadioButton selected dot |
| Destructive | `theme.colors.error` | Log out action label and confirmation Snackbar; "Couldn't load jobs" error banner text |

**Accent reserved for:**
1. Detail screen "Accept job" button (`mode="contained"`)
2. Onboarding "Save area" button (`mode="contained"`)
3. Active bottom-tab indicator (Paper Tabs default)
4. RefreshControl spinner color
5. Selected `RadioButton` dot

Do NOT apply accent to: Card surfaces, Card text, Settings rows, Appbar background, status badges (see below).

**Status badge color (PENDING-only feed):**
- Phase 8's feed only ever shows `PENDING` jobs (server filters; client also drops non-PENDING on WS update). A status badge is therefore visually redundant and is **omitted** from the Feed Card and the Detail screen. If a future phase reintroduces multi-status feeds, badge colors should be defined then.

**Destructive reserved for:**
- "Log out" button label color in Settings (text-only, not contained)
- Top-level error banner text on the Feed when initial fetch fails
- Snackbar background tint stays at Paper default `inverseSurface` (not `error`) per Paper convention â€” the 409 Snackbar is informational ("Job already taken"), not destructive

## Interaction Flows (the 5 primary flows)

### Flow 1: First-time onboarding
1. User logs in (Phase 3 flow)
2. `NavigationGuard` (extended) detects authenticated user + no `service_area` in SecureStore â†’ `router.replace('/(app)/onboarding')`
3. User picks an area â†’ taps "Save area" â†’ SecureStore write â†’ `router.replace('/(app)/(tabs)/feed')`
4. Feed mounts and fetches `GET /api/jobs?cityArea=<area>`

### Flow 2: Tap a Card â†’ push to detail
1. User taps a Card on the Feed tab
2. `router.push('/(app)/jobs/' + id)` â€” Stack push, animated slide-in
3. Detail screen fetches `GET /api/jobs/:id`
4. Body renders with category as heading, three labeled fields, "Accept job" button at bottom

### Flow 3: Acceptance success path (200)
1. On Detail screen, user taps "Accept job"
2. Button enters loading state
3. `POST /api/jobs/:id/accept` with body `{ version: jobDto.version }`
4. 200 response â†’ `router.back()` returns to Feed
5. Within milliseconds, the WS `JOB_UPDATED` event (status=ACCEPTED) arrives â†’ Feed drops the job from its list
6. No additional UI confirmation (the disappearance is the confirmation; matches Phase 3 "no confirm dialog" mobile aesthetic)

### Flow 4: Acceptance conflict path (409)
1. On Detail screen, user taps "Accept job"
2. Button enters loading state
3. `POST /api/jobs/:id/accept` returns 409
4. Snackbar shows "Job already taken" (Paper default surface, `bodyLarge` text)
5. Detail screen calls `router.back()` immediately (does not wait for Snackbar to auto-dismiss â€” Paper Snackbar persists across navigation if anchored at root layout; if anchored locally, the Feed shows its own Snackbar via shared state OR the 409 Snackbar is shown on the Feed after navigation â€” implementation choice for executor, both are acceptable as long as the user sees the message)
6. Feed triggers refetch via `useFocusEffect` or a refetch trigger from a shared ref/context

**Decision for executor:** Show the Snackbar **on the Feed screen after navigating back**, anchored to the Feed's root view. Use a small shared piece of state (e.g., a ref in a context, or pass a navigation param) to signal "show 409 toast". This keeps Snackbar lifecycle simple and avoids cross-screen Snackbar-anchor complexity.

### Flow 5: Pull-to-refresh
1. User pulls down on the Feed FlatList
2. `RefreshControl.refreshing = true`
3. Re-issue `GET /api/jobs?cityArea=<area>`
4. On response, replace list state, set `refreshing = false`
5. On error, set `refreshing = false`, show top error banner

### Flow 6 (passive): WebSocket job-removed
1. Another provider accepts a job currently in this user's feed
2. Backend broadcasts `JOB_UPDATED` (status=ACCEPTED)
3. Feed's WS handler runs the rule: status !== PENDING â†’ filter out by id
4. Job disappears with no animation requirement (Paper/RN default)

## Accessibility Contract

| Concern | Rule |
|---------|------|
| Minimum touch target | 44pt on every Card, Button, ListItem, RadioButton row, Tab bar item, Snackbar action. Enforced via `contentStyle.minHeight = 44` on Buttons and `style.minHeight = 44` on Card wrappers. |

| Tab bar | Paper Tabs sets `accessibilityRole="tab"` and `accessibilityState.selected` automatically. |
| Snackbar | Paper Snackbar uses `accessibilityLiveRegion="polite"` by default; copy is short enough to be announced once. Verify on iOS VoiceOver during executor smoke test. |
| Focus order on Detail | Heading (category) â†’ Description label/value â†’ Timeframe label/value â†’ Area label/value â†’ "Accept job" button. Use default DOM-equivalent order (no explicit `accessibilityElementsHidden` overrides needed). |

| Loading announcement | When the "Accept job" button enters loading state, `accessibilityState.busy = true` triggers VoiceOver "in progress" hint. |
