# Plan 01: Database Schema & Types Foundation

## Objective

Establish the database schema and shared types for reviews. This foundation enables all subsequent review submission, approval, and display features.

**Purpose:** Define the data model and type contracts that all review APIs and UI will depend on.

**Output:** Schema migration (reviews table with enums) + exported TypeScript types in shared package.

## Context

All existing Drizzle ORM patterns follow the structure in `apps/web/lib/db/schema.ts` — use pgEnum for enums, serial for IDs, and timestamps with defaults.

Review data structure per [CONTEXT.md](11-CONTEXT.md):
- **Client→Provider ratings:** Communication, Quality, Punctuality (1–5 each)
- **Provider→Client ratings:** Payment Reliability, Communication Clarity, Professionalism (1–5 each)
- **Approval:** `approvedAt` nullable (null = pending, Date = approved)
- **Uniqueness:** One review per (jobId, reviewerId) pair

### Task 2: Create reviews table in schema

**File:** `apps/web/lib/db/schema.ts`

Add reviews table export following existing Drizzle patterns. The table must include:

**Fields:**
- `id`: serial, primary key
- `jobId`: integer, foreign key to jobs(id), not null
- `reviewerId`: integer, foreign key to users(id), not null (who wrote the review)
- `revieweeId`: integer, foreign key to users(id), not null (who is being reviewed)
- `reviewType`: 'client' | 'provider' — denormalizes who the reviewer is
- `clientRating`: JSON object with communication, quality, punctuality (1-5 each) OR three separate int columns
- `providerRating`: JSON object with paymentReliability, communicationClarity, professionalism (1-5 each) OR three separate int columns
- `text`: varchar, nullable (optional review text)
- `photoUrl`: varchar, nullable (Vercel Blob Storage URL)
- `approvedAt`: timestamp, nullable (null = pending approval; set to now() when admin approves)
- `createdAt`: timestamp, default now()
- `updatedAt`: timestamp, default now(), onUpdate now()

**Constraints:**
- `unique(jobId, reviewerId)`: enforces one review per job per person per RATING-02
- Foreign keys: (jobId, reviewerId, revieweeId)
- Index on `(revieweeId, approvedAt)` for efficient "approved reviews for user" queries

Position table after jobs table or at end. Use existing naming conventions (snake_case columns, camelCase JS accessors).

**Verification:**
```bash

grep "unique.*jobId.*reviewerId" apps/web/lib/db/schema.ts
grep "index.*revieweeId.*approvedAt" apps/web/lib/db/schema.ts
```

## Threat Model

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client→API | Untrusted client input (rating values, text, photo) |
| API→Database | Trusted (enforced by Drizzle schema constraints) |
| Admin→Review Approval | Admin user with role check must approve before visibility |

### STRIDE Register

| Threat ID | Category | Component | Mitigation |
|-----------|----------|-----------|-----------|
| T-11-01 | Tampering | revieweeId field | API must verify reviewer identity via JWT; never trust client-supplied reviewerId |
| T-11-02 | Information Disclosure | pending reviews | approvedAt remains nullable; queries filter WHERE approvedAt IS NOT NULL |
| T-11-03 | Denial of Service | review submission | Rate limit reviews per user per job; enforce unique(jobId, reviewerId) at schema level |
| T-11-04 | Tampering | rating values | Range validation (1-5) at API layer; schema INT columns; low exploit value |
