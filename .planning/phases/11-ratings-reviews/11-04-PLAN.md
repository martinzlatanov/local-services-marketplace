# Plan 04: Admin Dashboard & Real-Time WebSocket Integration

## Objective

Build admin review moderation dashboard and integrate real-time WebSocket notifications. Admins approve pending reviews; providers receive instant notification when their review goes public.

**Purpose:** Enable content moderation and provide real-time feedback to providers.

**Output:** Admin dashboard + approval workflow + WebSocket integration.

## Context

Builds on Wave 3 (review UI and data flow). Requires existing WebSocket infrastructure from Phase 6.

WebSocket event contract:
```typescript
{
  type: 'review_approved',
  reviewId: number,
  revieweeId: number,
  reviewerName?: string
}
```

## Tasks

### Task 1: Create admin review moderation dashboard page

**File:** `apps/web/app/admin/reviews/page.tsx`

Create admin-only page for reviewing and approving pending reviews.

**Route:** `/admin/reviews` or `/dashboard/admin/reviews`

**Auth:**
- Check authenticated user has role = 'admin'
- Redirect to /unauthorized if not admin

**Page Structure:**
1. **Header:**
   - Title: "Pending Reviews" or "Review Moderation Queue"
   - Optional filter/search: By provider name, job ID, date range
   - Sort: "Newest first" (default)
   - Badge: Show count of pending reviews (e.g., "5 pending")
2. **Main content:**
   - Grid or list of ReviewApprovalCard components
   - Each card shows: reviewer name, reviewee name, rating categories, review text excerpt, photo thumbnail, date
   - Actions: "Approve" and "Reject" buttons on each card
3. **Empty state:** "No pending reviews" if queue empty

**Behavior on approve:**
- Admin clicks "Approve"
- Card shows loading state
- POST /api/reviews/approve with {reviewId}
- On success: Card disappears
- On error: Show toast with error message
- Pending count badge decrements

No checkpoint needed for this task; admin can approve multiple reviews. However, add checkpoint task (Task 4) for manual verification.

### Task 2: Create ReviewApprovalCard component

**File:** `apps/web/components/ReviewApprovalCard.tsx`

- `reviewerName: string`

**UI Structure:**

**Behavior:**
- On "Approve": POST /api/reviews/approve, show loading, hide card

- Error: Show toast if API fails

Match existing admin card/table styles.

### Task 3: Add WebSocket broadcast to POST /api/reviews/approve

**File:** `apps/web/app/api/reviews/approve/route.ts`

Update the approval endpoint to broadcast a WebSocket event.

**Changes:**
1. After updating approvedAt to now():
   - Fetch reviewee's user ID from review record
   - Emit WebSocket event: 'review_approved'
   - Event payload: {reviewId, revieweeId, reviewerName}
2. Use existing WebSocket utility in the project:
   - If `broadcast()` exists: broadcast('review_approved', {reviewId, revieweeId})
   - If `emit()` exists: io.to(revieweeId).emit('review_approved', ...)
   - If custom emitter: eventEmitter.emit('review_approved', {...})
3. Return 200 + updated review (same as before)

**Error handling:**
- WebSocket emit is fire-and-forget
- Log errors but return 200 to client
- Approval persists even if notification fails

**Verification:**
```bash

grep "emit.*review_approved" apps/web/app/api/reviews/approve/route.ts
grep "websocket|WebSocket" apps/web/app/api/reviews/approve/route.ts
```

### Task 4: Add client-side WebSocket listener for review_approved event

**File:** `apps/web/lib/websocket.ts`

Add or update the WebSocket listener to handle 'review_approved' events.

**Context:** WebSocket utility likely already exists from Phase 6. Executor should ADD to existing listeners.

**Changes:**
1. If utility exports `addEventListener` or `on()`:
   ```typescript
   export function onReviewApproved(callback: (data: {reviewId, revieweeId}) => void) {
     socket.on('review_approved', callback);
   }
   ```
2. Alternatively, if utility exports listener object:
   ```typescript
   export const listeners = {
     ...existing,
     review_approved: (data) => {
       // Trigger profile refresh
     }
   }
   ```

**Usage in ReviewDisplay:**
- Add useEffect that subscribes to 'review_approved'
- When event received with matching revieweeId: refetch GET /api/reviews?userId=X&approved=true
- Update local state with new reviews

**Behavior:**
- Provider visits own profile
- Another user's review approved by admin
- WebSocket broadcasts 'review_approved' event to provider
- Provider's profile automatically refreshes review list (no page reload)
- New review appears in ReviewDisplay in real-time

**Error handling:**
- If WebSocket not connected: graceful degradation
- Listener logs errors but doesn't crash

**Verification:**

## Threat Model

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin→Approval API | Trusted: verified admin role via JWT |
| Server→WebSocket broadcast | Trusted: internal event system |
| WebSocket→Client | Partially trusted: client validates revieweeId before refresh |

### STRIDE Register

| Threat ID | Category | Component | Mitigation |
|-----------|----------|-----------|-----------|
| T-11-14 | Privilege Elevation | /admin/reviews page | Admin role check on page load; redirect non-admins |
| T-11-15 | Tampering | approval endpoint | JWT identifies approver; audit log recommended (future) |
| T-11-16 | Information Disclosure | pending queue | /admin/reviews accessible only to admins |
| T-11-17 | Denial of Service | approval endpoint | Each approval atomic; no bulk operations |
