# Plan 02: Review APIs — Submission, Query, & Approval

## Objective

Implement review submission, approval, and query APIs. These endpoints enable clients/providers to submit reviews, admins to approve them, and profiles to display approved reviews with aggregate ratings.

**Purpose:** Connect review data model to HTTP layer. Enable all downstream UI flows.

**Output:** Three API routes with CRUD + approval logic, proper error handling, and photo validation.

## Context

Builds on Wave 1 (schema + types). Uses existing JWT middleware and API patterns from `apps/web/app/api/jobs/` routes.

Review submission structure:
```typescript
{
  jobId: number;
  clientRating?: { communication: 1-5, quality: 1-5, punctuality: 1-5 };
  providerRating?: { paymentReliability: 1-5, ... };
  text?: string; // max 1000 chars
  photoUrl?: string; // Vercel Blob URL
}
```

## Tasks

### Task 1: Implement POST /api/reviews — review submission endpoint

**File:** `apps/web/app/api/reviews/route.ts`

Create POST handler accepting review submission from authenticated client/provider.

**Flow:**
1. Extract JWT user (userId, role: 'client' | 'provider')
2. Parse request body: {jobId, clientRating?, providerRating?, text?, photoUrl?}
3. Determine review direction:
   - If user is job creator (client): reviewType = 'client', clientRating must exist
   - If user is job provider: reviewType = 'provider', providerRating must exist
   - Reject if role doesn't match job ownership (403)
4. Validate job exists and status is COMPLETED
5. Validate rating values: each category 1–5
6. Validate text length: max 1000 chars if provided
7. Validate photoUrl (if provided):
   - File size < 5 MB
   - Content-type is image/jpeg, image/png, or image/webp
   - If invalid, return 400
8. Try inserting review into db
9. Catch unique constraint violation → return 409 with "Review already submitted"
10. Return 201 + {id, createdAt, approvedAt: null}

**Error Responses:**
- 400: Invalid rating values, invalid photo, text too long
- 403: User not participant in job
- 404: Job not found
- 409: Review already submitted for this job
- 500: Database error

**Verification:**
```bash
grep "export.*POST" apps/web/app/api/reviews/route.ts
grep "409" apps/web/app/api/reviews/route.ts
grep "COMPLETED" apps/web/app/api/reviews/route.ts
```

### Task 2: Implement GET /api/reviews — query and profile endpoints

**File:** `apps/web/app/api/reviews/route.ts`

Create GET handler in the same route.ts file. Support two query patterns:

**Pattern 1:** `?jobId=X` (internal use — fetch reviews for specific job, all states)
- Optional: `?approved=true` filter to approved only
- Query: `SELECT * FROM reviews WHERE jobId = X [AND approvedAt IS NOT NULL]`
- Return: array of review objects

**Pattern 2:** `?userId=X&approved=true` (profile view — fetch approved reviews for user)
- Query: `SELECT * FROM reviews WHERE revieweeId = X AND approvedAt IS NOT NULL ORDER BY approvedAt DESC`
- Compute aggregate ratings per category (e.g., avg of communication across all reviews)
- Return: `{reviews: [...], averageRating: {communication: 4.2, quality: 4.5, ...}}`

No auth required for GET ?userId= (public profile view).
Auth required for GET ?jobId= (internal job detail view).

**Error Responses:**
- 400: Missing or invalid query params
- 403: User not authorized to view job reviews
- 404: User or job not found

**Verification:**
```bash
grep "jobId.*WHERE" apps/web/app/api/reviews/route.ts
grep "approvedAt.*IS NOT NULL" apps/web/app/api/reviews/route.ts
```

### Task 3: Implement POST /api/reviews/approve — admin approval endpoint

**File:** `apps/web/app/api/reviews/approve/route.ts`

Create POST handler for admin to approve pending reviews.

**Flow:**
1. Extract JWT user, verify role = 'admin'
2. Parse request body: {reviewId}
3. Fetch review by id, verify approvedAt is null
4. Update: set approvedAt to now()
5. Fetch updated review + details
6. Return 200 + updated review
7. **TODO for Wave 4:** Emit WebSocket 'review_approved' event to revieweeId

**Error Responses:**
- 400: Missing or invalid reviewId
- 403: User not admin
- 404: Review not found
- 409: Review already approved

**Verification:**
```bash

grep "admin.*403" apps/web/app/api/reviews/approve/route.ts

```

### Task 4: Implement GET + DELETE /api/reviews/[id] — detail and self-delete

**File:** `apps/web/app/api/reviews/[id]/route.ts`

Create GET and DELETE handlers for individual review routes.

**GET /api/reviews/[id]:**
- Fetch review by id
- If approvedAt is null, only allow access to reviewer or admin (403 otherwise)
- Return full review object
- Return 404 if not found

**DELETE /api/reviews/[id]:**
- Allow only review author (reviewerId = authenticated userId)
- Only allow deletion of pending reviews (approvedAt = null)
- Return 403 if already approved
- Delete row, return 204 No Content

These enable editing/deleting drafts before approval.

**Error Responses:**
- 403: User not reviewer or review already approved
- 404: Review not found
- 204: Success (no content)

**Verification:**

## Threat Model

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client→API (review submission) | Untrusted: rating values, text, photoUrl |
| Admin→Approval API | Trusted: verified admin role |
| Public→GET /reviews?userId | Partially trusted: public query, backend validates userId |

### STRIDE Register

| Threat ID | Category | Component | Mitigation |
|-----------|----------|-----------|-----------|
| T-11-05 | Tampering | revieweeId | API derives from job record, never trusts client input |
| T-11-06 | Repudiation | review deletion | Only pending reviews deletable by author; approved reviews immutable |
| T-11-07 | Information Disclosure | pending reviews | GET /reviews/[id] with approvedAt=null accessible only to reviewer or admin |
| T-11-08 | Information Disclosure | photo file | Validate photoUrl is Vercel Blob; size/type checked before storage |
| T-11-09 | Denial of Service | review submission | unique(jobId, reviewerId) prevents duplicates |
